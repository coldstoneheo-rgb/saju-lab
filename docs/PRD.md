# Saju Lab PRD

Last updated: 2026-05-14

## 1. Product Summary

Saju Lab은 절기 기반 사주 계산 결과를 바탕으로 사용자가 자신의 성향, 커리어, 재무 흐름을 이해하도록 돕는 설명형 리포트 앱이다. 핵심 경험은 단문 운세가 아니라 근거, 불확실성, 행동 제안을 함께 보여주는 report-first 분석이다.

초기 제품은 한국어를 우선 지원하고, 이후 영어 및 한/영 병행 리포트로 확장한다. 해석은 오락과 자기이해를 중심에 두되 커리어/재무 결정지원 성격을 갖기 때문에 단정적 예언, 투자/의학/법률 자문처럼 보이는 표현은 금지한다.

## 2. Goals

| ID | Goal | Success Signal |
| --- | --- | --- |
| G1 | 설명형 리포트가 첫 화면 경험의 중심이 된다 | 사용자가 입력 후 바로 구조화된 리포트를 탐색할 수 있다 |
| G2 | 커리어/재무 섹션이 실용적 결정지원으로 동작한다 | 경향, 리스크, 행동 제안이 분리되어 표시된다 |
| G3 | 무료 일간/월간 요약으로 재방문 동기를 만든다 | 핵심 키워드 3개와 프리미엄 리포트 연결이 제공된다 |
| G4 | 계산/해석의 불확실성을 투명하게 표시한다 | 출생시간 미상, 추정 정보, 신뢰도 배지가 명시된다 |
| G5 | 규칙 기반 코어로 AI 없이도 기본 리포트가 생성된다 | `rules-only` 모드가 테스트 가능한 결과를 반환한다 |

## 3. Non-Goals

- MVP에서 의학, 투자, 법률 판단을 대신하지 않는다.
- MVP에서 결제, 계정, 개인 데이터 동기화까지 완성하지 않는다.
- 검증되지 않은 절기/사주 결과를 확정 데이터로 표시하지 않는다.
- AI 생성 문장을 단독 진실처럼 노출하지 않는다.

## 4. Target Users

### Casual Explorer
- 가벼운 사주 리포트를 읽고 싶은 일반 사용자
- 쉬운 설명, 무료 요약, 모바일 친화적 흐름을 기대한다.

### Decision Support User
- 커리어 전환, 재무 계획, 장기 흐름을 참고하고 싶은 사용자
- 근거, 리스크, 행동 제안을 분리해서 보고 싶어 한다.

### Global Expansion User
- 한국어 외 언어에서도 동아시아 명리학을 현대적 설명으로 이해하려는 사용자
- 용어 현지화와 비신비적 톤이 중요하다.

## 5. MVP Scope

### Input
- 양력 생년월일 입력
- 출생시간 입력 또는 시간 미상 선택
- 성별 선택
- 타임존 기본값 `Asia/Seoul`

### Calculation
- 절기 기준 연주/월주 계산
- 60갑자 순환 기반 일주/시주 계산
- 출생시간 미상 시 시주 생략 및 신뢰도 하향
- KASI 등 공개 절기 데이터 기반 골든 픽스처 검증

### Report
- `ReportV1` 스키마 기반 리포트 생성
- 섹션: Overview, Personality, Career, Finance, Yearly Outlook, Monthly Highlights, Action Suggestions, Transparency
- 각 섹션은 확정 정보와 해석 정보를 가능한 한 분리한다.
- 모든 리포트에는 정보/오락 목적 및 비자문 disclaimer를 포함한다.

### Free Summary
- 일간/월간 키워드 3개
- 상세 리포트로 이어지는 링크
- 리포트 생성 전에도 제품 가치를 보여주는 간단한 미리보기

### i18n
- 한국어 키를 우선 확정한다.
- 리포트 문구는 key-value 기반으로 관리한다.
- 영어 확장 시 동일 스키마와 키 구조를 유지한다.

## 6. User Flow

1. 사용자가 홈에서 리포트 생성 CTA를 선택한다.
2. 입력 화면에서 생년월일, 출생시간, 성별을 입력한다.
3. 출생시간 미상인 경우 신뢰도 영향 안내를 확인한다.
4. 계산 결과와 규칙 기반 해석으로 리포트가 생성된다.
5. 사용자는 커리어/재무, 연간/월간, 행동 제안, 투명성 섹션을 탐색한다.
6. 무료 요약 화면에서 상세 리포트로 재진입할 수 있다.

## 7. Functional Requirements

| ID | Requirement | Priority |
| --- | --- | --- |
| FR-01 | 생년월일, 출생시간, 성별, 타임존을 입력받는다 | P0 |
| FR-02 | 출생시간 미상 모드를 지원한다 | P0 |
| FR-03 | 절기 기준 연주/월주 계산 규칙을 구현한다 | P0 |
| FR-04 | `calculatePillars(input)` 인터페이스를 제공한다 | P0 |
| FR-05 | `generateReportV1(input)` 인터페이스를 제공한다 | P0 |
| FR-06 | `ReportV1` 스키마와 JSON 직렬화를 지원한다 | P0 |
| FR-07 | 리포트에 confidence와 missing data note를 표시한다 | P0 |
| FR-08 | 한국어 i18n 키 기반 UI/리포트 텍스트를 사용한다 | P1 |
| FR-09 | 무료 월간/일간 요약 화면을 제공한다 | P1 |
| FR-10 | 리포트 snapshot 테스트를 제공한다 | P1 |

## 8. Non-Functional Requirements

- 정확성: 절기 경계, 입춘 기준 연주 변경, 경계 동시각 규칙을 테스트로 고정한다.
- 투명성: 확정 정보와 추정 정보를 UI와 데이터 모델 모두에서 분리한다.
- 안전성: 금융/의학/법률 자문으로 오해될 문구를 금지한다.
- 재현성: 같은 입력은 같은 `ReportV1` 결과를 생성해야 한다.
- 확장성: 계산 코어, 리포트 생성, UI, i18n을 패키지 단위로 분리할 수 있어야 한다.

## 9. Release Criteria

- 핵심 입력 5개 이상이 골든 픽스처 테스트를 통과한다.
- 출생시간 있음/없음 케이스가 모두 리포트로 생성된다.
- 리포트 모든 섹션이 `ReportV1` 스키마에 맞게 표시된다.
- disclaimer와 transparency 섹션이 누락되지 않는다.
- 한국어 키 파일에 MVP 화면과 리포트 필수 문구가 포함된다.
- README, PRD, architecture, roadmap, interface spec이 서로 충돌하지 않는다.

## 10. Open Questions

- 일주 계산 기준 라이브러리를 직접 구현할지, 검증된 오픈소스/데이터셋을 채택할지 결정이 필요하다.
- 절기 데이터 캐시를 앱 내부에 포함할지, 빌드 타임에 생성할지 결정이 필요하다.
- 프리미엄 리포트의 결제/계정 범위는 MVP 이후 별도 PRD로 분리한다.
- AI 보조 해석을 언제 도입할지, 규칙 기반 결과와 어떻게 구분 표기할지 정책이 필요하다.
